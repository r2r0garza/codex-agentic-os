- Completed: `execute_next_step` previously called `routing_policy.resolve(...)` *before* dispatching the step to `RUNNING`, so a `ValueError` from no-eligible-provider resolution propagated out uncaught — the step stayed `QUEUED` forever with no recorded reason, and would have crashed `worker.run_worker`'s poll loop (its `except (ApprovalRequiredError, ContextReferencesUnresolvedError)` clause does not catch a bare `ValueError`). Reachable in practice: a capability valid at step-creation time can have no eligible provider at dispatch under a narrower operator-supplied `--provider-preference` policy. Fixed by catching that `ValueError`, still dispatching the step to `RUNNING` via the existing `start_next_step` path (`provider_route=None`), then failing it through the existing `fail_step_from_error` helper — the same path already used for adapter-call failures after dispatch, so no new state-transition or history code was needed. `RunStep.failure_kind`/`retry_eligible` already classify any failed provider-message step with a string `error`/`error_type` output as `StepFailureKind.DEFINITE`, so retry-eligibility came for free; `worker.run_worker` needed no change since `execute_next_step` now returns a terminal `(step, run)` tuple instead of raising. Also changed `start_next_step`/`transition_step` to derive the `step_started` history entry's `required_capability` from the step's own message rather than only from a resolved `ProviderRoute`, so a failed-routing dispatch still records the requested capability (`resolved_provider`/`resolved_model`/`routing_reason` stay `null`); fixed-provider steps are unaffected. Documented the failure behavior in DEVELOPMENT.md's `run history` section. Added Plan 0091.
